# 学生项目进展日志

为当前项目维护一份清晰、实用、可长期积累的中文进展日志。

## 工作目标

把零散的日常工作整理成适合学生项目使用的进展记录，方便后续继续用于：
- logbook
- 周报
- 中期报告
- 结题报告
- 答辩材料
- Git 版本记录

默认使用项目根目录下的 `progress.md`。

## 模式选择

根据用户意图，优先选择以下三种模式之一：

1. 日常记录模式
   - 用于整理当天进展
   - 适合“帮我记今天做了什么”“更新今日进展”这类请求
2. 周报模式
   - 用于整理本周完成情况、问题和下周计划
   - 适合“帮我写这周总结”“整理本周项目进展”这类请求
3. logbook 模式
   - 用于生成更正式、可直接改写进 logbook 的记录
   - 适合“帮我写 logbook”“整理成实验日志风格”这类请求

当用户要求周报模式或 logbook 模式时，读取 [references/modes.md](C:\Users\15376\.codex\skills\student-progress-log\references\modes.md) 中对应模板后再写。

## 标准流程

1. 读取项目根目录下已有的 `progress.md`。
2. 如果用户要求“总结以前做了什么”或“补历史记录”，先检查项目中的关键线索：
   - 文件夹名称
   - 文件修改时间
   - Git 提交记录
   - 关键代码目录
   - 报告、PPT、实验输出、阶段性文档
3. 根据用户需求选择模式：日常记录、周报、logbook。
4. 用中文整理为结构化记录。
5. 如果信息是从文件直接确认的，就正常写。
6. 如果信息是根据时间和文件内容推断的，必须用保守表达，例如：
   - “可判断”
   - “推断”
   - “应包括”
   - “说明当时可能已经开始……”
7. 如果用户同时希望备份：
   - 更新 `progress.md`
   - 运行 `git status`
   - 视情况 `git add`
   - 写清楚提交信息
   - 在仓库已配置远程时执行 `git push`

## 默认记录格式

日常记录模式优先使用下面这个格式：

### YYYY-MM-DD
- 主要目标：
- 已完成：
- 关键文件：
- 问题 / 卡点：
- 下一步：
- Git 备份：

周报模式和 logbook 模式的详细模板见 [references/modes.md](C:\Users\15376\.codex\skills\student-progress-log\references\modes.md)。

## 写作要求

- 全程使用中文，除非用户明确要求英文。
- 语气务实、清楚，不写空话套话。
- 优先写“今天实际做了什么”，而不是泛泛而谈。
- 内容要适合学生后续直接拿去改成正式材料。
- 不夸大进展，不虚构成果。
- 对无法直接确认的历史信息，必须把“事实”和“推断”区分开。

## 历史总结要求

当用户要求补写前期阶段进展时：

1. 按时间段或阶段整理，不要堆零散文件名。
2. 优先识别这些阶段：
   - 前期资料收集
   - 初期报告准备
   - 中期阶段推进
   - baseline 搭建与改造
   - 算法实现与调试
   - 实验复现与结果整理
   - Git / 文档规范化
3. 结合目录名、文件时间、实验输出、报告文档来写。
4. 如果证据不足，不要写得太满。

## 常用提交信息风格

优先写简洁中文提交信息，例如：
- `更新 2026-03-11 项目进展`
- `补充前期阶段总结`
- `整理本周实验记录`
- `添加中文项目进展日志`
- `更新干扰抑制实验进展`
- `补充本周周报与下一步计划`
- `整理 logbook 风格实验记录`

## 何时需要多读一些文件

在这些情况下，先多检查项目内容再写：
- 用户要“总结以前都做了什么”
- 用户要“整理本周进展”
- 用户要“写成 logbook”
- 项目目录里存在多个日期阶段
- 有多个版本目录，例如 `1.20`、`2.14`、`3.7`
- 用户希望把记录写得能直接用于报告材料

## 输出原则

- 默认直接更新 `progress.md`
- 如有必要，同时给用户一段简短总结
- 如果用户明确要求周报或 logbook，可以在 `progress.md` 中追加对应小节，或先输出草稿给用户确认
- 如果当天确实改了代码或文档，提醒或协助执行：

```bash
git status
git add .
git commit -m "更新当天项目进展"
git push
```
